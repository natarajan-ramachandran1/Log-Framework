# WxLog Framework - Enterprise Logging Solution

## Overview

The **WxLog Framework** is an enterprise-grade logging solution for webMethods Integration Server, designed to provide unified, structured logging across all integration services.

### Evolution

The WxLog framework has been serving as a reliable logging utility for webMethods Integration Server for several years. In its latest iteration, the framework has been significantly enhanced to meet modern observability requirements by incorporating **Elastic Common Schema (ECS)** compliance. This upgrade enables organizations to leverage industry-standard log formats and integrate seamlessly with the Elastic Stack ecosystem for advanced log management, analysis, and monitoring.

### Package Components

- **WxLog** (v1.0) - Core logging engine built on Apache Log4j2
- **WxLogClient** (v1.0) - Client utilities and convenience services

The enhanced framework now implements the **Elastic Common Schema (ECS)** JSON layout, providing structured, standardized logging that integrates directly with Elasticsearch, Kibana, and other ECS-compatible platforms.

## Key Features

- **Log4j2-based engine** - Fast, efficient, and industry-standard
- **Automatic initialization** - Starts on server startup via `wx.log.services:initialize`
- **MDC support** - Contextual logging with Mapped Diagnostic Context
- **Message catalogs** - Externalized, per-package message definitions
- **Automatic service detection** - Uses service namespace as logger name
- **ECS JSON Layout** - Elastic Common Schema compliance
- **Convenience services** - logStart, logComplete, logError
- **Asynchronous logging** - High performance with LMAX Disruptor
- **Data Model integration** - Maps application data model to ECS fields
- **Custom ServiceAppender** - Invoke IS services on log events (configured in Log4j2 XML)

| `destination.service.name` | Target system | Application data |

| `user.id` | User identifier | Auto-generated by WxLog |
| `service.name` | Service name | Auto-generated by WxLog |
| `http.request.method` | HTTP method (GET, POST) | Application data |
| `http.response.status_code` | HTTP status code | Application data |
| `host.hostname` | Server hostname | Auto-generated by WxLog |
| `process.pid` | Process ID | Auto-generated by WxLog |

## Configuration

### Log4j2 Configuration File

**Location:** `<IntegrationServer>/packages/WxLogClient/config/Logging.xml`

**Example Configuration:**

```xml
<?xml version="1.0" encoding="UTF-8"?>
<Configuration status="debug">
  <Appenders>
    <!-- Console with ECS Layout -->
    <Console name="Console">
      <EcsLayout />
    </Console>

    <!-- Rolling File Appender with ECS Layout -->
    <RollingFile name="ESB" 
                 fileName="logs/ESB/api_ecs.log" 
                 filePattern="logs/ESB/%d{yyyy-MM}/api_ecs-%d{MM-dd-yyyy}-%i.log">
      <EcsLayout />
      <Policies>
        <TimeBasedTriggeringPolicy />
        <SizeBasedTriggeringPolicy size="250 MB" />
      </Policies>
      <DefaultRolloverStrategy>
        <Delete basePath="logs/ESB" maxDepth="2">
          <IfFileName glob="api_ecs-*.log"/>
          <IfLastModified age="3d"/>
        </Delete>
      </DefaultRolloverStrategy>
    </RollingFile>
  </Appenders>

  <Loggers>
    <logger name="ESB" level="DEBUG" additivity="false">
      <AppenderRef ref="ESB"/>
    </logger>
    <Root level="INFO">
      <AppenderRef ref="Console"/>
    </Root>
  </Loggers>
</Configuration>
```

### Configuration Best Practices

1. **Log Rotation:** Configure size limit per file based on your requirements
2. **Retention:** Set retention period based on compliance and storage needs
3. **Format:** ECS JSON for Elastic Stack compatibility
4. **Compression:** Configure based on storage and performance requirements
5. **Async Logging:** Enabled via LMAX Disruptor for performance

```xml
<File name="FILE" class="ch.qos.logback.core.FileAppender">

  </encoder>

```

```xml
<appender name="DB" class="com.softwareag.wx.log.ServiceAppender">

  <user>Administrator</user>
</appender>
```

```xml
<!-- Set Package log level -->

<!-- Send a logger to a different appender -->
<logger name="WxLog.wx.log" level="TRACE" >
  <appender-ref ref="FILE2"/>
</logger>
```

## Service Template Integration

Every business service should include three log points:

```
┌─────────────────────────────────────┐
│  Business Service Flow              │
├─────────────────────────────────────┤
│  1. logStart                        │
│     ↓                               │
│  2. Business Logic                  │
│     ↓                               │
│  3. logComplete (success path)      │
│     OR                              │
│     logError (error path)           │
└─────────────────────────────────────┘
```

### IData to ECS Message Field Mapping

All service input and output IData objects are automatically converted to JSON and mapped to the ECS `message` field. The framework processes the following IData fields:

| IData Field | Description | Mapped to ECS Field |
|-------------|-------------|---------------------|
| `logDetail/Input` | Service input parameters and request data | `message` (as JSON) |
| `logDetail/Output` | Service output parameters and response data | `message` (as JSON) |
| `logDetail/Error` | Error details including stack trace and error context | `message` (as JSON) |

**How it works:**
1. The logging services (`logStart`, `logComplete`, `logError`) accept IData objects
2. These IData objects are converted to JSON format
3. The JSON is stored in the ECS `message` field
4. Additional metadata (transaction.id, service.name, etc.) is stored in their respective ECS fields

### Implementation Example

#### Step 1: Log Service Start

**Service:** `WxLogClient.service.pub:logStart`

**Input:**
```javascript
logDetail: {
  Input: {
    // Your service input parameters
    customerId: "CUST-12345"
    requestType: "CREATE"
    // ... other input fields
  }
}
mdc: {
  transaction.id: "TXN-20251117-067"
  source.service.name: "SourceSystem"
  destination.service.name: "TargetSystem"
  event.type: "RequestSubmitted"
  // ... other application data fields
}
```

**Result:** The `logDetail/Input` IData is converted to JSON and stored in the `message` field of the log entry.

#### Step 2: Execute Business Logic

Your service implementation goes here.

#### Step 3: Log Completion or Error

**For Success - Service:** `WxLogClient.service.pub:logComplete`

**Input:**
```javascript
logDetail: {
  Output: {
    // Your service output parameters
    status: "SUCCESS"
    responseId: "RESP-98765"
    // ... other output fields
  }
}
mdc: {
  transaction.id: "TXN-20251117-067"
  source.service.name: "TargetSystem"
  destination.service.name: "SourceSystem"
  // ... response metadata
}
```

**Result:** The `logDetail/Output` IData is converted to JSON and stored in the `message` field of the log entry.

**For Error - Service:** `WxLogClient.service.pub:logError`

**Input:**
```javascript
logDetail: {
  Error: {
    // Error details
    errorCode: "ERR-001"
    errorDescription: "Validation failed"
    failedField: "customerId"
    // ... other error details
  }
}
mdc: {
  transaction.id: "TXN-20251117-067"
  error.type: "Input Validation"
  error.message: "Invalid or missing required fields"
  error.stack_trace: "<stack trace details>"
  // ... other error details
}
```

**Result:** The `logDetail/Error` IData is converted to JSON and stored in the `message` field of the log entry.

### Complete Example

```javascript
// At service start
logDetail/Input: {
  orderId: "ORD-12345"
  customerId: "CUST-67890"
  amount: 1500.00
}

// Logged as:
{
  "message": "{\"orderId\":\"ORD-12345\",\"customerId\":\"CUST-67890\",\"amount\":1500.00}",
  "transaction.id": "TXN-20251117-067",
  "service.name": "processOrder",
  // ... other ECS fields
}
```

## MDC (Mapped Diagnostic Context)

### Implicit MDC Fields

Automatically added by the framework:

- `host.hostname` - Server hostname
- `user.id` - Current user
- `trace.id` - Root context ID (distributed tracing)
- `process.pid` - Process ID
- `service.name` - Service name
- `service.namespace` - Service namespace
- `service.environment` - Environment (DEV/STAGE/PROD)

### Managing MDC

Use `wx.log.services:mdc` service:

#### Put Entries
```javascript
Input: {
  put: {
    transaction.id: "TXN-12345"
    customer.id: "CUST-789"
  }
}
```

#### Get Entries
```javascript
Input: {
  get: ["transaction.id", "customer.id"]
}

Output: {
  entries: {
    transaction.id: "TXN-12345"
    customer.id: "CUST-789"
  }
}
```

#### Remove Entries
```javascript
Input: {
  remove: ["transaction.id"]
}
```

#### Clear All
```javascript
Input: {
  clear: true
}
```

## Message Catalogs

### Creating Message Catalogs

Create `resources/messages_en_US.properties` in your package:

```properties
USER_LOGIN.text=User ${user.id} logged in successfully
USER_LOGIN.level=INFO

ERROR_DB_CONNECTION.text=Database connection failed: ${error.message}
ERROR_DB_CONNECTION.level=ERROR

TRANSACTION_COMPLETED.text=Transaction ${labels.transactionType} completed for ${labels.customerId}
TRANSACTION_COMPLETED.level=INFO
```

### Using Message Catalogs

Log using message ID:

```javascript
// Using wx.log.services:log
Input: {
  id: "USER_LOGIN"
  mdc: {
    user.id: "john.doe"
  }
}
```

The framework will:
1. Look up the message text from the catalog
2. Replace variables (e.g., `${user.id}`) with MDC values
3. Use the level defined in the catalog (or override if provided)

## Log Levels

| Level | Usage | Example |
|-------|-------|---------|
| **ERROR** | Error events that might still allow continuation | Database connection failed |
| **WARN** | Potentially harmful situations | Deprecated API usage |
| **INFO** | Informational messages highlighting progress | Service started, Transaction completed |
| **DEBUG** | Fine-grained debugging information | Request/Response payloads |
| **TRACE** | Finer-grained than DEBUG | Method entry/exit points |

### Setting Log Levels

**In Configuration File:**
```xml
<logger name="AppLogger" level="DEBUG" additivity="false">
  <AppenderRef ref="AppLogger"/>
</logger>
```

**At Runtime:**
Log levels can be adjusted without restart using Log4j2's JMX support.

## Best Practices

1. **Use appropriate log levels**
   - Reserve ERROR for actual errors
   - Use INFO for business events
   - Use DEBUG for detailed troubleshooting

2. **Include contextual information**
   - Always populate transaction.id
   - Add relevant identifiers using `labels.*` fields
   - Include source and destination systems

3. **Use message catalogs**
   - Centralize message definitions
   - Support internationalization
   - Maintain consistency

4. **Configure rolling policies**
   - Set appropriate file size limits
   - Define retention period based on requirements
   - Configure compression based on needs

5. **Use ECS layout**
   - Enables Elastic Stack integration
   - Structured JSON format
   - Standard field names

6. **Set logger names strategically**
   - Use package/folder/service hierarchy
   - Enable granular control
   - Support filtering in Kibana

7. **Monitor log file sizes**
   - Check disk space regularly
   - Review retention policies
   - Archive old logs if needed

8. **Asynchronous logging**
   - Enabled by default
   - High performance
   - Non-blocking

9. **Follow service template**
   - Always include logStart
   - Always include logComplete or logError
   - Never skip error logging

10. **Use custom labels consistently**
    - All custom fields must use `labels.*` prefix
    - Define meaningful field names
    - Document custom fields in your application

## Troubleshooting

### Logging Not Working

**Symptoms:** No log files created or logs not appearing

**Solutions:**
1. Verify WxLog package is enabled and loaded
2. Check configuration file exists: `packages/WxLogClient/config/Logging.xml`
3. Review Integration Server logs for initialization errors
4. Confirm `watt.wx.log.config` extended setting
5. Check file permissions on logs directory
6. Verify Log4j2 JARs are present
7. Check for Environment global variable

### Performance Issues

**Symptoms:** Slow service execution, high CPU usage

**Solutions:**
1. Verify async logging is enabled (LMAX Disruptor)
2. Adjust buffer sizes in configuration
3. Use separate appenders for different log levels
4. Review pattern layouts for optimization
5. Consider reducing log level in production
6. Check disk I/O performance

### Missing MDC Values

**Symptoms:** ECS fields showing null or empty

**Solutions:**
1. Ensure MDC values set before logging
2. Check thread context not cleared prematurely
3. Verify MDC keys match ECS field names
4. Review service template implementation
5. Check for thread recycling issues

### Log Files Growing Too Large

**Symptoms:** Disk space issues, large log files

**Solutions:**
1. Verify rolling policy is configured
2. Check retention policy settings
3. Ensure Delete strategy is active
4. Review log levels (reduce DEBUG in production)
5. Archive old logs to external storage

### ECS Fields Not Mapping

**Symptoms:** Fields not appearing in Elasticsearch

**Solutions:**
1. Verify EcsLayout is configured
2. Check Filebeat configuration
3. Review Elasticsearch index mapping
4. Confirm ECS version compatibility
5. Test with sample log entry

## Integration with Elastic Stack

The ECS JSON layout enables seamless integration with Elastic Stack:

### Architecture

```
┌─────────────────────┐
│ Integration Server  │
│ (WxLog Framework)   │
└──────────┬──────────┘
           │ Writes ECS JSON
           ▼
┌─────────────────────┐
│ File System         │
│ logs/ESB/*.log      │
└──────────┬──────────┘
           │ Monitors
           ▼
┌─────────────────────┐
│ Filebeat            │
│ (Log Shipper)       │
└──────────┬──────────┘
           │ Ships logs
           ▼
┌─────────────────────┐
│ Elasticsearch       │
│ (Storage & Index)   │
└──────────┬──────────┘
           │ Queries
           ▼
┌─────────────────────┐
│ Kibana              │
│ (Visualization)     │
└─────────────────────┘
```

### Benefits

- **No Logstash required** - Logs already in ECS format
- **Direct ingestion** - Filebeat to Elasticsearch
- **Standard fields** - ECS-compliant field names
- **Easy querying** - Consistent structure across services
- **Rich visualizations** - Pre-built Kibana dashboards

### Sample Filebeat Configuration

```yaml
filebeat.inputs:
  - type: filestream
    id: ecs-json-input
    enabled: true
    paths:
      - /usr/share/filebeat/data/*.log
    parsers:
      - ndjson:
          keys_under_root: true
          overwrite_keys: true
          add_error_key: true

processors:
  - decode_json_fields:
      fields: ["message"]
      target: ""
      overwrite_keys: true
      add_error_key: true

setup.ilm.enabled: false
setup.ilm.check_exists: false
setup.template.enabled: true
setup.template.name: "filebeat"
setup.template.pattern: "filebeat-*"
setup.template.fields: "${path.config}/fields.yml"

output.elasticsearch:
  hosts: ["http://opensearch_logging:9201"]
  username: "<user>"
  password: "<pass>"
  ssl.verification_mode: none
```

**Note:** Adjust paths, hosts, and credentials according to your environment.
